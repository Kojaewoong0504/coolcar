# 시원칸 쾌적 칸 추천 재설계 v1

- 작성일: 2026-06-29
- 상태: 설계 확정 후보
- 범위: 자연어/음성 입력 제외, API 비용 최소화, 혼잡도/냉방/사용자 제보/환승 동선 기반 재설계

## 1. 결정 사항

### 자연어/음성 입력 제외

이번 재설계에서 자연어 입력, OpenAI STT, Realtime Voice Agent는 제외한다.

이유:

1. API 비용 예측이 어렵다.
2. 추천 critical path가 느려질 수 있다.
3. 출발지/목적지/생활 패턴/음성 데이터 개인정보 부담이 커진다.
4. 현재 핵심 차별화는 입력 방식보다 `쾌적 칸 추천 정확도`다.

향후 사용자가 충분히 늘고 수익/예산 구조가 생긴 뒤, 텍스트 자연어 입력부터 재검토한다.

## 2. 경쟁앱 혼잡도 표현 방식

국내 지하철/지도 앱의 혼잡도 표현은 대체로 4단계다.

| 단계 | 일반 라벨 | 주로 쓰이는 색상 | 생활 설명 |
|---|---|---|---|
| 1 | 여유 | 파랑 또는 초록 | 통로 여유, 움직이기 쉬움 |
| 2 | 보통 | 초록 또는 노랑 | 서서 이동 가능 |
| 3 | 주의 | 노랑 또는 주황 | 이동 시 부딪힘 가능 |
| 4 | 혼잡 | 빨강 | 이동 어려움 |

서비스별 확인:

- 또타지하철/TMAP: 칸별 혼잡도, 일부 실시간 + 일부 예측/통계, 초록/노랑/주황/빨강 계열.
- 지하철종결자: 파랑/초록/노랑/빨강 계열, 여유/보통/주의/혼잡 표현.
- 카카오맵 공항철도: 여유/보통/주의/혼잡 4단계, 객차별 실시간 혼잡도.
- 코레일 지하철톡: 코레일 구간 칸별 혼잡도 확대/시범 흐름.

시원칸은 브랜드상 `여유=파랑`을 기본으로 한다.

```txt
파랑: 쾌적/여유
초록: 무난
주황: 주의
빨강: 혼잡/피하기
회색: 데이터 부족
```

## 3. API 비용/요금 구조 판단

### 확정 가능한 사실

1. SK OpenAPI의 `지하철 혼잡도 / PUZZLE-SUBWAY_v2` 상품은 무료/유료 요금 구조가 있다.
   - 공개 조사 기준 Free는 매우 작은 월 한도이며, Basic은 건당 과금 구조다.
   - 기업고객은 별도 문의 구조.

2. TMAP 대중교통 상품의 통계성 열차 혼잡도도 무료 한도 + 유료 단가 구조가 있다.

3. 서울교통공사 공공 혼잡도 데이터는 무료 공개 데이터가 있으나, 파일 기반/30분 단위 평균이며 실시간 칸별 혼잡도와 다르다.

4. 카카오맵 공항철도, SKT-서울교통공사-TMAP, SKT-코레일 사례는 공개적으로 협력/MOU/데이터 결합 구조로 설명된다.

### 확정할 수 없는 것

- 지하철종결자, 또타지하철, 카카오맵, 코레일지하철톡 등이 공개 요금표 그대로 비용을 내는지는 공개 출처만으로 확정할 수 없다.
- 대형 앱들은 단순 외부 API 결제보다 기관/기업 간 제휴, 업무협약, 데이터 교환, 내부 정산 구조일 가능성이 있다.

### 시원칸 전략

시원칸은 초기부터 유료 API 대량 호출 구조로 가면 위험하다.

따라서:

1. 서버 캐시 필수.
2. 공공/정적/제보 데이터 우선.
3. SK/TMAP 유료 API는 선택적 보강 데이터로만 사용.
4. 호출은 `역+노선+요일+시간대` 기준으로 묶어 캐시한다.
5. 무료 한도에 의존한 제품 약속 금지.

## 4. 제품 포지셔닝

시원칸은 `혼잡도 표시 앱`이 아니라 `쾌적 칸 추천 앱`이다.

핵심 문장:

> 노선도 말고, 지금 어디 탈지만 알려줘.

보조 문장:

> 덜 덥고, 덜 붐비고, 덜 걷는 칸을 추천해요.

## 5. 데이터 계층

시원칸 추천은 다음 데이터 계층을 합성한다.

| 계층 | 데이터 | 비용/위험 | 사용 방식 |
|---|---|---|---|
| 정적 규칙 | 약냉방칸, 양끝 시원함, 중앙부 더움 | 낮음 | 기본 냉방 점수 |
| 공공 통계 | 서울교통공사 혼잡도 CSV 등 | 낮음, 실시간 아님 | 시간대/역/방향 혼잡 보정 |
| SK/TMAP | 칸별 혼잡도 API | 유료/한도 위험 | 캐시된 보강 데이터 |
| 사용자 제보 | 온도/혼잡/자리/고장 체감 | 콜드스타트/어뷰징 | 최신 체감 보정 |
| 동선 데이터 | 빠른 환승/하차/출구 근접 | 수집 필요 | 환승/하차 점수 |

## 6. 추천 엔진 v1

각 칸의 최종 점수는 다음 요소로 계산한다.

```txt
comfortScore =
  coolingScore
+ crowdingScore
+ feedbackScore
+ transferScore
+ confidenceScore
- uncertaintyPenalty
```

### 6.1 coolingScore

- 시원함 우선:
  - 약냉방칸 감점
  - 일반냉방칸 가점
  - 칸 내부 양끝 위치 안내
- 덜 추움 우선:
  - 약냉방칸 가점
  - 중앙부/약냉방 쪽 안내
- 균형:
  - 약냉방칸은 사용자 선호에 따라 중립 또는 약한 감점

### 6.2 crowdingScore

- 데이터가 있으면 칸별 혼잡도를 `여유/보통/주의/혼잡`으로 변환한다.
- 칸별 데이터가 없으면 역/방향/시간대 평균 혼잡도를 전체 열차 기본값으로 사용한다.
- 사용자 제보가 충분하면 칸별 보정한다.

### 6.3 feedbackScore

사용자 버튼 제보만 받는다. 자연어 입력은 받지 않는다.

제보 항목:

```txt
온도: 더워요 / 적당해요 / 시원해요 / 추워요
혼잡: 여유 / 보통 / 혼잡 / 매우 혼잡
상태: 냉방 약함 / 냉방 강함 / 냄새 / 고장 의심 / 자리 있었음
```

TTL:

```txt
0~10분: 강하게 반영
10~30분: 중간 반영
30분 이후: 약하게 반영 또는 통계화
```

### 6.4 transferScore

- 목적역/환승역/출구 정보가 있으면 반영한다.
- 목적역이 없으면 낮은 가중치로만 반영한다.
- `빠른 환승`이 너무 혼잡하거나 약냉방이면 균형 추천에서 밀릴 수 있다.

## 7. 추천 모드

P0에서는 3개 모드만 제공한다.

1. **시원함 우선**
   - 더위 많이 타는 사용자.
   - 일반냉방 + 덜 혼잡 + 양끝 위치 우선.

2. **덜 추움 우선**
   - 냉방을 싫어하는 사용자.
   - 약냉방칸 + 중앙부/바람 덜 직접적인 위치 우선.

3. **균형 추천**
   - 기본값.
   - 냉방/혼잡/동선을 함께 고려.

## 8. 결과 UI

### 8.1 상단 결론

```txt
오늘은 1번 칸이 좋아요
덜 붐비고, 일반냉방 쪽이에요.
```

### 8.2 1~10칸 지도

기존 차량 칸 지도는 반드시 유지한다.

- 추천 칸: 파랑 + `추천`
- 대안 칸: 초록 + `대안`
- 피하기: 빨강/주황 + `피하기`
- 데이터 부족: 회색 + `확인중`

### 8.3 칸 상세

칸을 누르면 소비자 언어로 설명한다.

```txt
1번 칸
- 덜 붐비는 편이에요
- 일반냉방 쪽이에요
- 내릴 때 동선은 보통이에요
- 칸 안에서는 양쪽 끝이 더 시원해요
```

금지 표현:

```txt
cache hit
provider score
raw confidence
통계 API 오류
```

## 9. P0 범위

이번 재설계의 P0는 다음만 포함한다.

1. 자연어/음성 입력 제외.
2. 추천 모드 3개.
3. 약냉방칸 정적 규칙 반영.
4. 칸 내부 위치 팁 반영.
5. 혼잡도 4단계 표현.
6. 사용자 버튼형 제보 수집.
7. 1~10칸 지도 개선.
8. 추천 TOP 2~3개 표시.
9. 데이터 출처/신뢰는 내부 보존, 화면은 소비자 문구로 표현.
10. SK/TMAP API는 캐시 우선, 실패 시 공공/정적/제보 fallback.

## 10. P1 이후

1. 공공 혼잡도 CSV 자동 적재.
2. 빠른 환승/하차/출구 가까운 칸 DB.
3. 사용자 제보 신뢰도 모델.
4. 시간대별 제보 패턴화.
5. 실시간 도착 열차별 추천.
6. Toss 인앱 저장/공유/출퇴근 알림.

## 11. 수용 기준

P0 완료 조건:

1. 추천 결과에서 1~10칸 지도가 항상 표시된다.
2. 자연어/음성/OpenAI 호출이 없다.
3. 추천 모드 3개가 결과에 반영된다.
4. 약냉방칸이 시원함 우선/덜 추움 우선에서 반대로 작동한다.
5. 혼잡도는 `여유/보통/주의/혼잡` 4단계로만 표시한다.
6. 사용자 제보 버튼이 있고, API 응답/저장 smoke가 통과한다.
7. SK/TMAP API 실패 또는 미설정 시에도 앱이 동작한다.
8. 화면에는 기술어/원시 점수/API 비용 관련 문구가 노출되지 않는다.
